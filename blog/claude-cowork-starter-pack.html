Starter

先建立一個能跑的起手式，不要先追求全家桶

根據原始文章標題與 preview，這份 starter pack 的重點是「setup」。所以第一階段只做一件事：讓 Claude Cowork 每次開工都拿到同一組上下文、同一套插件入口、同一條執行路線。

最小可用組合

  • 一個主要用途，例如內容營運、客戶跟進、會議準備或研究整理。
  • 一組固定 context 檔，而不是每次重新打一段提示詞。
  • 一個主要插件，先把資料入口接好，不要同時裝十個。
  • 一條可以每天重複的 workflow，才能驗證是否真的省時間。

建議先做的 4 份 context

context/
├─ about-me.md
├─ business-overview.md
├─ brand-voice.md
└─ working-preferences.md

about-me.md
  - 你是誰、做什麼、主要客群
business-overview.md
  - 產品、服務、流程、輸出格式
brand-voice.md
  - 語氣、禁用詞、常用 CTA
working-preferences.md
  - 你喜歡的工作節奏、回報格式、交付標準

這組切法的好處是：人設、商業背景、品牌語氣、操作偏好彼此獨立。之後要調整時，只需要更新單一檔案，不必整份 system prompt 重寫。

什麼時候該升級成 Skill？

  • 同一段開場指令已經重複三次以上。
  • 這個任務有固定輸入格式與固定完成定義。
  • Claude 每次都做得差不多，但你還要補同樣的提醒。
  • 你希望新開一個 session 也能用同一套方式執行。

第一條 workflow 應該怎麼選

  • 選結果容易驗證的工作，例如整理待辦、寫文章初稿、會前研究。
  • 選資料邊界清楚的工作，不要一開始就做跨十個工具的總控台。
  • 選每週至少會重複兩到三次的工作，否則很難驗證效益。
  • 選可以保留人工審核的一段流程，不要直接把高風險決策全自動化。
Stack

把 Cowork 拆成 4 層：Context、Plugins、Skills、Cadence

這樣拆的理由很直接：如果所有問題都塞進 prompt，你永遠不知道到底是缺資料、缺工具、缺流程，還是缺回報節奏。分層後，每一層都可以獨立調整。

讓 Claude 知道你是誰、做什麼、什麼才算交付完成

這層解決的是理解問題。沒有 context，再好的插件也只是在更快地做錯事。

給 Cowork 資料入口，而不是把資料手貼進對話

原始文章標題特別提到 plugins。對多數使用者來說，先裝一個 Productivity 類插件，再視需要串 Notion、Calendar、Drive 或通訊工具，就足夠形成工作閉環。

把重複工作 productize，而不是每次 improvising

Corey 在後續公開貼文提到，他停止寫一次性 prompt，改成 reusable skills。這代表流程被明確定義，輸出更穩定，也更容易交接。

固定開工節奏，決定這套系統到底省不省時間

如果每天都用不同方法交辦，同一套技能也會失效。Cadence 的本質是讓人與 AI 之間的 handoff 變得可預期。

判斷哪一層出問題的速查表

  • 回答很空泛：多半是 Context 不夠，不是模型不夠強。
  • 知道要做什麼但沒資料：先補 Plugins 或資料來源。
  • 每次做法都不同：該把流程升級成 Skill。
  • 有時快有時慢：多半是沒有固定 Cadence，handoff 品質不穩。

先找故障層級，再決定要改哪裡。這比無止盡換 prompt 有效得多。

Workflow

每天照著同一條工作流跑，Claude 才會像系統而不是聊天框

Corey 公開描述過一個三方分工模型：他本人負責最終決策，AI agent 負責草稿、研究與技術執行，人類助理負責訊息發送、排程與檔案管理。這代表 Cowork 的價值不在「取代所有人」，而在於讓任務分派更清楚。

01

先做晨間 brief，不要直接丟散亂任務

每天開始前，用 5 到 10 分鐘把今天的主要目標、限制、截止時間與優先順序講清楚。這一步不是形式，而是避免 agent 後續猜測。

  • 今天最重要的 1 到 3 件事是什麼。
  • 哪件事需要你本人決策，哪件事可以先由 AI 起草。
  • 有沒有特定資料夾、文件、日曆事件或素材要優先讀。
02

每個任務都補齊 Why / Done / Access

這是 Corey 在公開貼文裡提到最有價值的任務欄位組合。沒有這三欄，AI 只能自己猜；有了這三欄，任務就更接近可執行 SOP。

  • Why：為什麼要做，幫助模型抓到上下文與優先順序。
  • Done：完成定義是什麼，避免只做「看起來差不多」。
  • Access：需要哪些連結、檔案、登入或資料位置。
03

把高頻工作封裝成 reusable skill

一旦某個任務重複出現，就不要再從零交辦。直接寫成 skill，讓 Claude 知道輸入格式、執行順序、例外處理與最終輸出長什麼樣子。

  • 文章工作流：研究 → 大綱 → 初稿 → 標題變體 → 素材清單。
  • 會議工作流：背景研究 → 人物摘要 → 討論風險 → 問題清單。
  • 待辦整理：分類 → 指派 → 草擬回覆 → 建立 follow-up。
04

保留人工審核點，尤其是外發內容與行程

Claude 可以準備，但不代表應該直接替你送出所有東西。對外訊息、排程、承諾、付款與高風險決策，仍應保留人類確認節點。

05

每日收尾只記錄下一班要接什麼，不要寫成流水帳

如果你希望跨 session 穩定銜接，收尾紀錄要足夠讓下一次開工的人或 agent 立刻知道「現在做到哪、還缺什麼、下一步先做哪件」。

Examples

可以先複製的 3 種 starter workflow

不要一開始就做複雜代理人群組。先挑一個資料來源清楚、輸出容易驗證、每週重複度高的工作，這樣最容易真的跑起來。

Content Sprint

適合創作者或顧問。用固定 context 與品牌語氣，讓 Claude 先完成研究摘要、文章大綱、標題備選與發佈 checklist。

Meeting Prep

適合業務、創辦人與顧問。把日曆事件、來賓資料、過往紀錄與目標 outcome 串起來，在會前自動生成 briefing note。

Task Triage

適合個人經營者與小團隊。將待辦清單按 Why / Done / Access 補齊，並預先標記哪些任務交給 AI，哪些保留給人處理。

Pitfalls

大多數人不是不會用 Claude，而是錯把 setup 問題當成模型問題

原始文章 preview 直接點出問題本質：很多人裝完 Claude Cowork，摸十分鐘就放棄。真正卡住的通常不是模型，而是這四個地方。

常見失敗點

  • 沒有固定 context，導致每次都像第一次合作。
  • 一次接太多插件，結果資料來源彼此衝突或難以維護。
  • 把所有事情都丟給 Claude，卻沒有定義完成標準。
  • 任務沒有 human review 節點，最後不敢真的交辦。

修正原則

  • 先讓第一條 workflow 每週穩定跑兩次以上，再擴充。
  • 先把資料入口與任務格式標準化，再追求更高自主性。
  • 先把重複工作 productize，再討論 agent network。
  • 讓 AI 先做準備與草稿，決策與承諾仍由人完成。
我需要一開始就寫很多 skills 嗎？

不需要。先挑一個高頻任務，把輸入、步驟與完成定義寫清楚。第一個 skill 的目的不是完整，而是驗證你自己的工作流是否能被標準化。

如果我已經有很多 prompt，下一步是什麼？

先找出重複率最高且最容易驗證輸出的那一條，把它升級成 skill。再把相關背景資訊拆成 references 或 context 檔，不要繼續把所有內容堆在同一個 prompt。

這套方法適合個人工作者還是團隊？

兩者都適合。個人工作者可以把它當成單人營運系統；小團隊則可以把任務 ownership、handoff 與 review 節點標準化，降低資訊落差。

先做出第一條可重複工作流

如果你今天只帶走一件事，那就是：先建立一條你每週真的會跑的流程，把它拆成 context、plugin、skill、cadence 四層。當這條線穩定後，再去擴充第二條、第三條。
