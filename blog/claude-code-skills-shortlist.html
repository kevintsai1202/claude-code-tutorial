Criteria

Skill 留不留，不看酷不酷，只看能不能穩定省時間

這篇的核心洞見很直接：技能越多不一定越強。當 skill 類型彼此重疊、命中率不穩、輸出差異太大，整個系統只會越來越難用。

保留標準

  • 高頻任務會穩定觸發，而且能持續省時間。
  • 輸入場景清楚，不容易和其他 skill 重疊。
  • 輸出品質足夠一致，能進入日常工作流。
  • 維護成本合理，不需要你每週重寫一次。

該刪除的訊號

  • 你已經忘記這個 skill 何時該用。
  • 它和其他 skill 功能相近，常常互搶觸發。
  • 每次用還是要補很多口頭說明。
  • 它讓 Claude 更常走錯方向，而不是更穩定。

判斷問題一

這個 skill 是否對某個高頻場景形成明確優勢？如果沒有，通常只是收藏，不是工作資產。

判斷問題二

這個 skill 是否比「直接說一句話」更省事？如果沒有，它可能不值得長期維護。

判斷問題三

它是否會讓 Claude 更難選擇正確路徑？如果會，代表 skill 數量或邊界已經失控。

Groups

真正值得留下的 10 類 skill，通常都屬於這幾個群組

原文標題提到最後只留下 10 個。這類保留清單通常不是 10 個零散名字，而是 10 種高價值能力。下面用分組方式整理，方便你依照自己場景取用。

研究與摘要類

適合高頻資訊整理工作，像是文章研究、競品分析、會前 briefing、重點摘要。

  • 輸入固定、輸出容易驗證
  • 很容易形成可重複模板

內容與格式類

例如品牌語氣、固定欄位輸出、長文改寫、資料整理。這類 skill 很常用，但也最容易互相重疊。

  • 要明確切分不同輸出場景
  • 避免「萬用內容 skill」過肥

工具與轉換類

把 CLI、腳本或資料轉換工作包成穩定入口，例如 CSV 整理、檔案轉換、命令流程封裝。

  • 最適合有明確輸入輸出 contract 的任務
  • 能大幅降低重複操作成本

流程與交接類

像是會議準備、任務 handoff、context log、專案節奏維護。這類 skill 的價值在於讓協作可持續，而不是只做好單次輸出。

  • 對長期專案最有價值
  • 能明顯降低 session 斷裂成本
Maintenance

Skill 清單需要像產品一樣維護，而不是只會增加不會刪除

很多人的 skill 體驗變差，不是因為個別 skill 做不好，而是整體 skill 庫從來沒有做過整理。真正有效的做法是持續修剪。

01

先列出目前真正常用的場景

不是看你裝了什麼，而是看你一週內真的重複做了哪些工作。場景比 skill 名字重要。

02

把 skill 對應到場景，不要一個場景掛三四個技能

如果一個需求對應太多 skill，Claude 的選擇成本會上升，人自己也會搞不清楚該用哪一個。

03

每隔一段時間移除低命中、低收益的 skill

保留「真的會用、真的有效」的 skill，刪掉只會增加噪音的項目，整體體驗反而會更好。

04

用 references 或拆分方式降低技能肥大

如果某個高價值 skill 開始變得過重，應該拆分或做 progressive disclosure，而不是讓它無限膨脹。

Pitfalls

Skill 爆量後最常出現的 3 個問題

一旦 skill 太多，問題通常不是「找不到功能」，而是「選不到正確功能」。這會直接降低 Claude 的穩定性與可預測性。

互搶觸發

多個 skill 都覺得自己適用，導致啟動錯誤或結果不一致。

維護失控

技能一多，更新成本暴增，最後沒有任何一個人確定哪個版本才是有效版本。

使用者混亂

連你自己都不確定該用哪個 skill，代表整個 skill 系統已經超出可管理範圍。

是不是 skill 越多，Claude 能力就越強？

不是。能力來自正確匹配與高品質執行，不來自選項數量。當選項過多，反而會增加模型與使用者的判斷成本。

我應該保留幾個 skill 才合理？

沒有固定數字，重點是每個 skill 都要對應一個穩定高頻場景，而且你清楚知道什麼時候該用它。若做不到，數量就太多了。

怎麼開始整理現有 skill 庫？

先列出最近一週最常做的 5 到 10 種任務，再把 skill 逐一對應。沒有對應高頻任務、或和別人高度重疊的 skill，優先檢討或移除。

保留真正高價值的 skill

這篇最重要的觀點是：Claude Code 的 skill 庫不是收藏櫃，而是工作系統的一部分。只留下真正有命中率、能省時間、能被維護的 skill，整體效率才會上升。
