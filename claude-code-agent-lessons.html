核心概念

動作空間設計：難在工具的取捨

打造 Agent 時，最困難的部分之一就是構建它的動作空間（action space）。Claude 透過 Tool Calling 行動，但工具的設計方式有很多種——從 bash、skills 到程式碼執行都有。

那麼，應該給 Agent 一個萬能工具，還是 50 個分工工具？

「把自己放進模型的位置：給你一道困難的數學題，你想要什麼工具？紙筆是最低需求，計算機更好，電腦最強大——但前提是你要知道怎麼用它。」
📄

紙筆（最低限度）

自由但雜亂，純靠手動計算，結果難以格式化與重現。

🖩

計算機（中等工具）

功能更強，但需要知道如何操作進階選項，才能發揮效用。

💻

電腦（最強能力）

速度最快、功能最強，但必須具備寫程式與執行程式碼的能力。

🔑

關鍵洞見

給 Agent 的工具必須符合其自身能力。要了解其能力，你需要像 Agent 一樣觀察：關注輸出、閱讀結果、不斷實驗。

01
第一課

改良問答能力：AskUserQuestion 工具

目標是提升 Claude 問問題的能力（稱為 elicitation，問答引導）。雖然 Claude 可以直接用純文字問問題，但我們發現回答那些問題耗費不必要的時間。

如何降低摩擦、提升用戶與 Claude 之間的溝通頻寬？我們嘗試了三種方法：

找到甜蜜點 · Finding the Sweet Spot
Modified Markdown Output model free but messy,
hard to format
✦ AskUserQuestion Tool structured + composable,
clear UI surface
ExitPlanTool Parameter plan already formed,
questions come too late
← No Structure Too Rigid →
01
✗ 失敗
修改 ExitPlanTool 參數

在 ExitPlanTool 加上一個問題陣列參數，讓計畫與問題一起傳回。雖然最容易實作，但這讓 Claude 感到困惑——它同時被要求產出計畫，又要針對計畫提問。

⚠ 問題：若用戶的回答與計畫內容衝突怎麼辦？Claude 需要呼叫 ExitPlanTool 兩次嗎？
02
✗ 失敗
修改輸出格式

修改 Claude 的輸出指令，讓它使用略經修改的 Markdown 格式來提問，例如用括號列出選項。前端再解析並呈現為 UI。Claude 雖能輸出，但不保證格式穩定——有時會補充額外句子、省略選項，或完全用不同格式。

⚠ 問題：Claude 的輸出是非確定性的，無法保證格式，導致解析困難。
03
✓ 成功
建立 AskUserQuestion 工具

最終方案是建立一個 Claude 可在任意時間點呼叫的工具，並在計畫模式中特別提示它使用此工具。工具觸發時，顯示一個 Modal 呈現問題，並阻斷 Agent 循環直到用戶回答。

✓ 優勢：強制 Claude 產出結構化輸出；讓用戶可在 Agent SDK 中組合使用；Claude 自然而然偏好呼叫它，輸出效果最好。
「即使是設計最好的工具，如果 Claude 不理解如何呼叫它，也沒有用。」
02
第二課

隨能力升級工具：從 Todos 到 Tasks

當我們剛推出 Claude Code 時，模型需要一個 Todo 清單來保持追蹤進度。Todos 在一開始寫好，隨工作進行勾選完成。

但隨著模型能力的提升，原本為了幫助模型的工具，反而開始限制它。

初期 · Early Stage
TodoWrite 工具

Claude Code 剛推出時，模型常常忘記自己要做什麼。我們提供了 TodoWrite 工具，讓 Claude 可以撰寫和更新 Todo 清單，並展示給用戶看。

⚙ TodoWrite Tool · 記錄與追蹤目標
調整期 · Adaptation
系統提醒每 5 回合觸發

即使有 Todo 清單，Claude 仍常忘記目標。我們每 5 回合插入系統提醒，強制告知 Claude 目前的任務。

🔔 System Reminders · 每 5 輪觸發
進化期 · Evolution
模型能力提升，提醒成了束縛

隨著模型進步，Claude 不再需要提醒——更糟的是，提醒讓它以為不能修改 Todo 清單。此外，Opus 4.5 在使用子 Agent 上大幅進步，但多個子 Agent 無法共享 Todo 清單。

⚡ 舊架構限制：單一 Agent, 無法跨 Agent 協調
現在 · Present
Task 工具取代 Todos

我們以 Task 工具替代 TodoWrite。Tasks 不僅是追蹤進度，更是幫助多 Agent 相互溝通——可設定依賴關係、跨子 Agent 分享更新、動態新增或刪除任務。

✓ Task Tool · 多 Agent 協作、依賴追蹤
「隨著模型能力提升，原本需要的工具可能反而會限制模型。必須持續重新審視舊有假設。」
03
第三課

設計搜尋介面：漸進揭露技術

搜尋工具對 Claude 來說至關重要——它讓 Claude 能夠主動建構自己的上下文（context）。

Claude Code 的演化路徑，展現了工具設計的逐步精煉：

1
RAG 向量資料庫

早期使用 RAG 提供上下文。雖然快速強大，但需要索引與設定，在不同環境中容易不穩定。最重要的是：Claude 是被動接收上下文，而非主動搜尋。

↓ 進化
2
Grep 工具

給 Claude 一個 Grep 工具，讓它可以搜尋程式碼庫，主動建構自己的上下文。這是一個重要的轉變：Claude 開始學會「看」了。

↓ 進化
3
Agent Skills + 漸進揭露

引入 Agent Skills，正式確立漸進揭露（Progressive Disclosure）模式：Claude 可以讀取 skill 文件，文件再參照其他文件，遞迴搜尋直到找到所需上下文。

↓ 最終形態
4
Claude Code Guide 子 Agent

針對「Claude 如何使用 Claude Code」這個特定需求，建立專屬子 Agent。當用戶問起相關問題時，主 Agent 呼叫此子 Agent，它有完整的文件搜尋指南，精確回傳所需答案——不多也不少。

核心哲學 · Core Philosophy

這是一門藝術，
不只是科學

如果你期待一套嚴格的工具設計規則，這篇文章無法給你。為模型設計工具，既是藝術，也是科學。它高度依賴你使用的模型、Agent 的目標，以及它所在的環境。
